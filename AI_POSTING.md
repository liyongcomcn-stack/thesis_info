一、选题名称（建议）

基于图文多模态学习的二手商品 AI 发布研究：类目属性提取、描述生成与业务效果评估

二、我想和导师先确认的核心结论

本论文拟研究在线二手交易平台中的 AI 发布 场景：基于用户上传的二手物品图片与输入标题，自动完成 类目提取、属性提取与商品描述生成，并进一步评估该能力是否能显著提升 B 端发布效率 与 C 端信息连接效率。

当前从工程与研究两方面看，整体路线是可行的：

模型层面：采用“单模态基线 + 多模态小模型主方案 + 大模型端到端强基线/描述生成辅助”的组合路线。
评估层面：分为离线评估与线上随机 A/B 测试两层。
业务层面：已有 B 端发布时长、发布成功等埋点数据，也可获取 C 端展示、点击、连接（电话/邮件等）数据，具备做业务效果识别的条件。
限制层面：当前没有最终成交数据，因此论文后半部分更适合聚焦 “发布效率提升” 与 “展示—点击—连接”链路改善，而不直接研究成交转化。
三、研究目标
3.1 总体目标

构建适用于二手交易发布场景的 AI 发布方案，并从模型效果与业务效果两个层面验证其价值。

3.2 具体目标
构建基于图片和标题的 类目提取模型。
在类目基础上构建 属性提取模型。
基于图片、标题、类目和属性生成 商品描述文本。
构建完整的 离线评价体系，评价准确率、完整率、通顺性、置信度与覆盖率。
通过 线上用户随机 A/B 测试，识别 AI 发布对发布时长、发布成功率、点击率和连接效率的真实影响。
四、研究问题
4.1 核心研究问题
RQ1：基于图片与标题的多模态模型，能否有效完成二手商品类目提取？
RQ2：在类目约束下，能否进一步准确抽取颜色、版本、成色、使用年限、损伤情况等属性？
RQ3：AI 生成的商品描述是否能够降低用户编辑负担，并提升描述质量？
RQ4：AI 发布是否能够显著缩短发布平均时长、提高发布成功率？
RQ5：AI 发布是否能够通过提升类目合理性、属性完整性和描述质量，进一步改善 C 端点击率与连接效率？
4.2 研究假设
H1：多模态模型在类目提取上的效果优于单模态模型。
H2：在类目条件约束下，多模态属性提取效果优于仅基于规则或单模态的方法。
H3：AI 生成描述能够降低用户编辑成本并提升描述质量。
H4：AI 发布能够显著缩短发布平均时长并提升发布成功率。
H5：AI 发布通过提升信息质量，间接提升点击率与连接效率。
H6：AI 发布效果在新老用户、头部长尾类目、图片质量高低之间存在异质性。

六、总体研究路线

七、模型方案设计

当前官方文档显示，Gemini API 提供多模态图像理解与结构化 JSON 输出能力；模型页列出了 Gemini 2.5 Flash-Lite（强调高速、低成本、多模态高频任务）和 Gemini 2.5 Pro（强调复杂任务能力）。阿里云百炼当前支持通过 OpenAI Chat Completion、OpenAI Responses 和 DashScope 调用千问，视觉理解侧官方推荐 Qwen3.5-Plus（效果优先）和 Qwen3.5-Flash（速度/成本优先）。这些能力都适合用于本研究中的多模态理解、结构化抽取和描述生成。

7.1 单模态与多模态模型总体分工
模块	方案定位	预期模型
文本单模态类目提取	基线	TF-IDF + LightGBM；MacBERT / Chinese RoBERTa
图片单模态类目提取	基线	EfficientNet-B3；ConvNeXt-Tiny
多模态类目提取	主方案	CLIP/SigLIP 图文向量 + MLP / LightGBM
属性提取	主方案	类目条件约束下的多任务分类模型；有序属性用 Ordered Logit/Ordered Probit
描述生成	生成模块	Gemini 2.5 Flash-Lite / Qwen3.5-Flash
端到端大模型	离线强基线	Gemini 2.5 Pro / Qwen3.5-Plus
低置信样本兜底	线上辅助	Gemini 2.5 Pro / Qwen3.5-Plus
7.2 单模态建模方案
（1）文本单模态

用于验证“仅靠标题文本能做到什么程度”。

预期模型：

经典基线：TF-IDF + LightGBM
深度学习基线：MacBERT / Chinese RoBERTa

用途：

类目提取基线
文本属性抽取基线
（2）图片单模态

用于验证“仅靠商品图片能做到什么程度”。

预期模型：

轻量 CNN 基线：EfficientNet-B3
更强视觉基线：ConvNeXt-Tiny

用途：

图片类目识别
视觉可见属性识别（颜色、外观类型、部分损伤）
7.3 多模态主方案
（1）类目提取主方案

预期模型：

CLIP / SigLIP 图文双塔编码
在融合后的 embedding 上训练 MLP / LightGBM 分类器

理由：

更适合结构化识别任务
可输出稳定概率，便于后续做置信度校准
推理延迟明显低于直接调用大模型端到端生成
（2）属性提取主方案

在类目已知或预测类目基础上，做条件式属性提取。

预期模型：

名义型属性（颜色、品牌、版本）：多分类模型
有序型属性（使用年限、成色、损伤等级）：Ordered Logit / Ordered Probit
开放型属性（局部瑕疵说明）：后接大模型补充生成

技术特点：

采用 类目先验约束
不同类目只预测其合法属性集合
降低无效属性与幻觉属性出现概率
7.4 大模型使用方式
（1）为什么不把大模型端到端作为线上主方案

当前已有实验观察显示，大模型端到端一次完整生成大约需要 5–10 秒，性能明显偏弱，不适合作为高频在线发布链路的唯一方案。因此，大模型在本研究中更适合作为：

离线强基线
描述生成模块
低置信样本兜底模块
（2）大模型的具体使用方式
Gemini 2.5 Flash-Lite / Qwen3.5-Flash：用于描述生成、高频轻量补全
Gemini 2.5 Pro / Qwen3.5-Plus：用于离线端到端强基线、低置信复杂样本兜底
（3）输出形式

通过 Structured Outputs / JSON Schema 约束大模型统一输出：

category
attributes
description

这样可以保证离线比较时字段级可对齐，便于公平评估。Gemini 官方文档明确支持基于 JSON Schema 的 structured outputs，适合结构化分类和数据抽取；阿里云百炼也支持 OpenAI 兼容接口，便于统一接入和实验。

八、描述生成模块的定位
8.1 模块目标

根据图片、标题、类目和属性，生成可直接用于发布的商品描述文本。

8.2 方案设计

优先采用：

结构模板约束
大模型 API 生成
必要时用户二次编辑
8.3 不优先做 LoRA 微调的原因

目前已有可直接调用的 Gemini 和通义千问 API，因此在论文主线中，描述生成部分优先采用 “API 直接调用 + 提示词/结构模板控制” 的方式。这样更符合当前资源条件，也能把研究重点放在：

模型效果比较
用户编辑成本变化
业务效果识别

如果后续资源允许，LoRA 可以作为扩展方向，用于提升平台特定风格一致性，但不是当前论文必须项。

十、线上随机 A/B 测试设计
10.1 实验目标

识别 AI 发布对 B 端发布效率、发布质量以及 C 端点击和连接行为的因果影响。

10.2 随机化方案

建议优先考虑：

按用户随机：避免同一用户在不同版本间来回切换
若产品实现受限，也可 按发布会话随机
分析时对用户维度做聚类稳健标准误处理

10.5 统计分析方法
主效应估计
发布成功率、点击率、连接率：Logit / Probit
发布时长：OLS / 对数 OLS
编辑次数：Poisson / 负二项回归
异质性分析

按以下维度分组：

新用户 vs 老用户
头部类目 vs 长尾类目
高质量图片 vs 低质量图片
中介效应分析

候选中介变量：

属性完整率
类目合理性
描述质量
用户编辑成本下降

研究链路：
AI 发布 → 信息质量提升 / 编辑负担下降 → 发布效率提升 / 点击连接改善

十一、论文的预期创新点
1. 场景创新

聚焦二手交易发布链路，不是研究通用商品理解，而是研究 “非标准化二手商品” 的类目、属性和描述生成问题。

2. 方法创新

提出 “单模态基线 + 多模态小模型主方案 + 大模型端到端强基线/兜底” 的组合研究框架，而不是直接把大模型端到端当作唯一方案。

3. 评估创新

不仅评估模型精度，还评估：

时延
成本
置信度
用户编辑负担
线上业务效果
4. 实证创新

依托真实线上随机 A/B 测试，识别 AI 发布对 B 端与 C 端链路的真实影响。

十二、预期风险与应对
风险 1：历史标签不完全可靠

应对：

对头部类目和重点属性做人工抽检
剔除异常样本
风险 2：大模型端到端结果不稳定

应对：

统一结构化输出
固定 prompt 和参数
对部分样本重复采样评估一致率
风险 3：没有成交数据

应对：

明确将研究重点放在“发布效率”和“展示—点击—连接”链路
将连接行为作为更接近交易意向的代理指标
风险 4：线上实验污染

应对：

尽量按用户级随机
分析时加入时间、类目、用户历史行为等控制变量
